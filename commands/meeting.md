# 미팅 진행 명령

> **위치**: 이 커맨드는 **범용 팀 토론**용이다. 아래 워크플로우 단계는 **전문화된 커맨드**를 우선 사용하라:
> - Feature 요구사항 수집 → `/sdlc:feature <name>`
> - PRD 생성 → `/sdlc:prd <name>`
> - 아키텍처 설계 → `/sdlc:architecture <name>`
>
> `/sdlc:meeting` 은 운영 이슈·스코프 협의·임의 토픽 등 위에 해당하지 않는 토론에 사용한다.

사용자가 `/meeting <참석자> | <주제> ` 형태로 호출했다.
전체 인자: `$ARGUMENTS`

## 1단계: 인자 파싱

인자를 `|` 구분자로 분리해라:
- 참석자 목록: 쉼표로 구분된 페르소나 키(예: `architect, qa, dev`)
- 주제: 논의할 질문 또는 결정 사항

인자가 부족하거나 형식이 틀렸다면 즉시 사용자에게 올바른 호출법을 안내하고 멈춰라:

```
올바른 호출법:
/meeting architect, qa, dev | OAuth 토큰 저장 전략 
```
산출물 경로: 최종 합의안을 저장할 상대 경로로 ${CLAUDE_PROJECT_DIR}/docs/meetings/meeting-<구분>-yyMMddHH-xx.md 형식
<구분>은 prd, plan, task 중 하나로 회의 내용을 기반으로 구분함
예를들어 prd 변경은 prd, plan 변경은 plan, 나머지는 task로 구분

## 2단계: 참석자 확인

`${CLAUDE_PLUGIN_ROOT}/agents/` 하위에 요청된 각 참석자의 페르소나 정의 파일(`<이름>.md`)이 실제로 있는지 `Glob`으로 확인해라. 없는 참석자가 있으면 경고하고 사용 가능한 페르소나 목록을 보여준 뒤 멈춰라.
참석자에는 적극적으로 대화에 참여하지는 않지만 필요한 의사결정을 내릴 수 있는 프로젝트 오너(CLAUDE.md 의 "프로젝트 오너" 섹션 참조)가 포함된다. 오너 섹션이 없으면 현재 사용자를 오너로 간주한다.
의사결정이 필요한 내용은 반드시 나에게 물어보고 대화를 계속할것.

## 3단계: 페르소나 로드

각 참석자의 `${CLAUDE_PLUGIN_ROOT}/agents/<이름>.md` 파일을 `Read`로 읽어 페르소나·전문성·어조를 파악해라. 이 내용은 이후 발언에 반영되어야 한다.

## 3.5단계: 관련 표준 문서 자동 로드 (중요)

`${CLAUDE_PROJECT_DIR}/docs/standards/` 하위에 팀 표준 문서가 있다. 미팅의 **참석자 구성**과 **주제**에 따라 관련 문서를 자동으로 `Read`해서 컨텍스트에 포함시켜라.

### 로드 규칙

1. **참석자 기반 매핑**:
   - `backend` 참석 → 주제에서 스택 키워드를 찾고 해당 `${CLAUDE_PROJECT_DIR}/docs/standards/backend/<스택>/` 파일들 읽기
   - `frontend` 참석 → `${CLAUDE_PROJECT_DIR}/docs/standards/frontend/*.md` 전부
   - `data` 참석 → `${CLAUDE_PROJECT_DIR}/docs/standards/database/*.md` 전부
   - `compliance` 참석 → 해당 스택의 `security.md`
   - `qa` 참석 → 해당 스택의 `testing.md` + `${CLAUDE_PROJECT_DIR}/docs/standards/frontend/testing.md` (프론트 주제면)
   - `architect` 참석 → 주제 관련 스택의 `structure.md`, `api.md` 와 `database/schema-design.md`

2. **주제 기반 스택 감지**:
   - "Spring", "Java", "JPA" 언급 → `backend/springboot/`
   - "Next.js", "TypeScript", "Node" 언급 → `backend/nextjs-typescript/` (+ 프론트면 `frontend/`)
   - "FastAPI", "Python", "Pydantic" 언급 → `backend/fastapi/`
   - 주제가 DB 관련 ("스키마", "쿼리", "인덱스", "마이그레이션") → `database/` 전부
   - 주제가 UI/UX 관련 → `frontend/`

3. **스택이 모호하면**:
   - 진행자(Nick)가 오프닝에서 먼저 확인:
     > "주제에 명시된 스택이 없네요. Spring Boot 기준으로 논의할까요, 아니면 다른 스택인가요? 기본은 **주제에 언급된 기술**을 따르겠습니다."
   - 사용자가 대답할 때까지 기다림 (또는 주제 본문에서 추정 후 명시)

4. **로드된 표준의 역할**:
   - 페르소나가 발언할 때 **권위 있는 기준**으로 인용
   - 예: "이건 `${CLAUDE_PROJECT_DIR}/docs/standards/backend/springboot/security.md` 의 JWT 규칙에 따르면 RS256 권장이라..."
   - 표준과 다른 제안을 할 땐 **예외 사유** 명시

5. **읽지 못하는 파일은 무시**:
   - `${CLAUDE_PROJECT_DIR}/docs/standards/` 가 없거나 일부 파일 없어도 오류 없이 진행
   - 단, 산출물 하단에 "표준 문서 부재로 일반 원칙 기반 논의"라고 명시

### 예시

주제: "Spring Boot에서 OAuth 토큰 저장 전략"
참석자: architect, backend, compliance

→ 자동 로드:
- `${CLAUDE_PROJECT_DIR}/docs/standards/backend/springboot/api.md`
- `${CLAUDE_PROJECT_DIR}/docs/standards/backend/springboot/security.md`
- `${CLAUDE_PROJECT_DIR}/docs/standards/backend/springboot/structure.md`
- `${CLAUDE_PROJECT_DIR}/docs/standards/database/schema-design.md` (토큰 저장 위치 논의 가능성)

## 4단계: 미팅 진행

당신(메인 Claude)이 **진행자(Facilitator) 역할**과 **각 참석자 역할**을 동시에 수행한다. 모두 하나의 컨텍스트 안에서 순차적으로 발화하되, 각 발언에 **명확한 헤더**를 붙여 누가 말하는지 표시한다.

### 진행 프로토콜

1. **오프닝** (진행자):
   - 주제를 한 문장으로 정리
   - 참석자 소개 (한 줄씩)
   - 첫 발언자 지명 (보통 주제와 가장 관련 있는 전문가)

2. **라운드 1 — 의견 개진** (2-4 턴):
   - 지명된 참석자가 자기 관점에서 의견 제시
   - 진행자가 다음 참석자에게 바톤 넘김
   - 각 참석자는 **이전 발언을 명시적으로 참조**해야 함 ("Winston 의견에 동의하는데, 테스트 관점에서는...")
   - 한 번은 반드시 이견이 나와야 한다. 모두 동의만 하면 토론이 아니다.

3. **라운드 2 — 트레이드오프 정리** (1-2 턴):
   - 진행자가 지금까지 나온 옵션을 표 형태로 정리해 제시
   - 참석자들이 남은 쟁점에 대해 추가 발언

4. **라운드 3 — 합의 도출** (1-2 턴):
   - 진행자가 합의 가능한 결정을 제안
   - 모든 참석자가 동의/반대 명시
   - 완전한 합의가 안 되면 "트레이드오프를 사용자에게 넘기고 결정 요청"으로 종료

5. **클로징** (진행자):
   - 결정 요약 (2-3줄)
   - 반대 의견 보존 (있다면)
   - 후속 조치 제안 (있다면)

### 발화 형식

각 발언은 반드시 다음 형식을 따른다:

```
**[이모지] [페르소나명] ([역할]):**
발언 내용 (3-6문단, 구체적·실용적)
```

예:
```
**🏛️ Winston (Architect):**
세 옵션의 트레이드오프를 먼저 깔고 가자...
```

### 상호작용 규칙

- 페르소나는 **자기 전문 분야에서만** 발화한다. 영역 밖 질문을 받으면 해당 전문가에게 넘긴다.
- 사용자가 중간에 개입하면 즉시 멈추고 사용자 답변을 기다린다.
- 같은 쟁점이 3 라운드 넘게 제자리면 진행자가 사용자 결정 요청으로 종료한다.
- 발화량은 균형 있게 — 한 명이 독점하지 않도록 진행자가 조율한다.

## 5단계: 산출물 작성

미팅이 끝나면 지정된 산출물 경로(`$3`)에 다음 형식의 파일을 `Write`로 생성한다:

```markdown
# 미팅 기록: {주제}

- **날짜**: {오늘 날짜 YYYY-MM-DD}
- **참석자**: {참석자 목록}
- **결정 상태**: 합의됨 | 부분합의 | 사용자결정필요

## 배경
{주제 왜 논의하게 됐는지, 1-2문단}

## 논의된 옵션

| 옵션 | 제안자 | 장점 | 단점 |
|------|--------|------|------|
| ...  | ...    | ...  | ...  |

## 합의 사항
- {채택된 결정}
- {채택 근거}

## 반대 의견 보존
- {채택되지 않은 의견과 이유 — 나중에 상황이 바뀌면 재검토할 수 있도록}

## 후속 조치
- [ ] {누가 무엇을 언제까지}

## 전체 대화 로그
{위의 미팅 대화를 그대로 append — 발언 순서·헤더 모두 보존}
```

## 6단계: 최종 보고

산출물 파일 생성 후 사용자에게 간단히 보고:

```
✅ 미팅 완료
참석자: {목록}
주제: {주제}
결정: {1줄 요약}
산출물: {경로}
```

## 주의사항

- 대화가 길어져도 하나의 Claude 응답 안에서 끝내라. 중간에 멈추고 사용자에게 "계속할까요?" 묻지 마라.
- 각 페르소나의 어조·전문성이 겹치지 않도록 차별화해라.
- 특정 기술이나 숫자에 대해 확신이 없으면 페르소나가 "그건 확인 필요"라고 말하게 해라. 가짜 단정 금지.
- 산출물 파일의 상위 디렉터리가 없으면 먼저 만들어라(`mkdir -p`).
- 기존 문서를 변경하는 경우 기존문서 변경을 수행하고, 기존문서명-history.md에 변경미팅참조와 주요내용을 반드시 기술해줘.
- 의사 결정이 필요한 부분이 있으면 가정하지 말고 대화 도중에라도 반드시 프로젝트 오너에게 물어보고 진행해줘.
