# Plan 진행 상황 확인

사용자가 `/status <Plan파일경로> [--update]` 형태로 호출했다.
전체 인자: `$ARGUMENTS`

예시:
```
/status ${CLAUDE_PROJECT_DIR}/docs/plans/email-verification.md
/status ${CLAUDE_PROJECT_DIR}/docs/plans/email-verification.md --update
```

## 1단계: 인자 파싱

- `$1`: Plan 파일 경로 (필수)
- `$2`: `--update` 플래그 (선택) — 있으면 Plan 파일에 스냅샷 섹션 갱신

파일 존재 확인. 없으면 중단하고 사용 가능한 Plan 목록 보여주기:

```bash
ls ${CLAUDE_PROJECT_DIR}/docs/plans/*.md 2>/dev/null
```

## 2단계: Plan 파일 파싱

Plan 파일을 `Read`로 읽고 다음 요소 추출:

### Story / Task 체크박스 상태

마크다운 체크박스 패턴 인식:
- `- [ ]` — 예정 (todo)
- `- [~]` — 진행중 (in_progress, 커스텀 관례)
- `- [!]` — 블로킹 (blocked, 커스텀 관례)
- `- [x]` 또는 `- [X]` — 완료 (done)

Epic/Story 헤더도 상태를 가질 수 있음:
- `### E1-S1: ... [todo|in_progress|blocked|done]`
- 또는 같은 체크박스 관례 적용

### DoD 체크리스트

각 Story의 `**DoD**:` 또는 `DoD:` 섹션 아래 체크박스 집계.

### 크기 정보

T-shirt size 태그 추출 (S/M/L/XL) — 각 상태별 규모 합산용.

### 담당 영역

`담당 영역: backend, dba` 형태에서 영역별 진행도 계산.

### 마일스톤

마일스톤 표에서 목표일·기준·해당 Story 목록 파싱.

## 3단계: 집계 계산

### 전체 진행률

```
완료 Story / 전체 Story
완료 Task / 전체 Task
완료 DoD 항목 / 전체 DoD 항목
```

단순 비율 외에 **크기 가중 진행률**도 계산:
- S=1, M=3, L=6, XL=10 점으로 환산
- (완료 점수 / 전체 점수) × 100%

### 상태별 분포

| 상태 | Story 수 | Task 수 | 크기 가중 |
|------|---------|---------|-----------|
| 예정 | ... | ... | ... |
| 진행중 | ... | ... | ... |
| 블로킹 | ... | ... | ... |
| 완료 | ... | ... | ... |

### 영역별 진행률

backend / frontend / dba / qa 등 각 영역의 진행률.

### 크리티컬 패스 진행

Plan의 "크리티컬 패스" 섹션에 언급된 Story ID들의 완료 여부 확인. 막힌 지점이 크리티컬 패스에 있으면 **⚠️ 경고** 표시.

### 블로킹 이슈

`- [!]` 체크박스가 있는 항목 모두 수집. 각 블로킹마다:
- 어느 Story·Task인지
- 담당 영역
- 크기

### 마일스톤 상태

각 마일스톤에 포함된 Story의 완료율 계산:
- 100% → ✅ 달성
- 부분 → 🟡 진행중 (퍼센트)
- 0% → ⬜ 미착수
- 목표일 지남 + 미완 → 🔴 지연

(목표일은 Plan 생성 날짜 + 주차로 계산. Plan의 "작성일" 메타데이터 사용)

## 4단계: 리포트 출력

사용자에게 다음 형식으로 응답:

```markdown
# 📊 진행 상황: {Plan 제목}

- **Plan 파일**: {경로}
- **조회 시각**: {현재 YYYY-MM-DD HH:mm}
- **작성일 기준**: {Plan 작성일로부터 N일 경과}

## 전체 진행률

```
Story:  ████████░░░░░░░  8/18  (44%)
Task:   ██████████░░░░░  28/52 (54%)
가중:   ███████░░░░░░░░  46%   (크기 기반)
```

## 상태별 분포

| 상태 | Story | Task |
|------|-------|------|
| ✅ 완료 | 8 | 28 |
| 🔄 진행중 | 3 | 7 |
| ⚠️ 블로킹 | 1 | 2 |
| ⬜ 예정 | 6 | 15 |

## 영역별 진행률

| 영역 | 진행률 |
|------|-------|
| backend | 60% (6/10) |
| frontend | 25% (2/8) |
| dba | 100% (3/3) |
| qa | 30% (2/6) |

## 크리티컬 패스 상태

✅ E1-S1 (DB 스키마) — 완료
✅ E1-S3 (토큰 검증) — 완료
🔄 E2-S1 (SES 통합) — 진행중 (담당: backend)
⬜ E2-S2 (가입 훅)
⬜ E3-S1 (검증 UI)

**다음 크리티컬 블로커**: E2-S1 (진행중). 지연 시 전체 일정 영향.

## ⚠️ 블로킹 이슈 (1건)

### E2-S1-T3: SES 프로덕션 access 요청 승인 대기
- 담당: backend + cloud
- 시작일: Plan 작성일 + 5일
- 블로킹 사유: AWS SES 샌드박스 → 프로덕션 전환 심사 중
- 예상 해제일: 불명
- 크리티컬 패스: **YES** — 1주 이상 밀리면 M2 마일스톤 영향

## 마일스톤 상태

| ID | 이름 | 목표 | 진행률 | 상태 |
|----|------|------|--------|------|
| M1 | 인증 기본 동작 | 주 2 | 100% | ✅ 달성 |
| M2 | 재전송 완성 | 주 4 | 60% | 🟡 진행중 (3일 남음) |
| M3 | 관리자 대시보드 | 주 6 | 0% | ⬜ 미착수 |

## 다음 액션 제안

- 🔴 E2-S1-T3 블로킹 해제 우선 추적 (AWS support 티켓 상태 확인)
- 🟡 E2-S1 완료 전까지 E2-S2 Mock으로 선행 가능 — frontend 트랙 유휴 방지
- 🟢 qa 진행률 낮음 — E1 완료 분의 테스트 backfill 고려
```

## 5단계: --update 플래그 처리

`$2`가 `--update` 라면, Plan 파일에 다음 섹션을 추가·갱신한다:

### Plan 파일 내 스냅샷 위치

Plan 파일 상단(메타데이터 아래)에 `## 📊 최근 상태 스냅샷` 섹션을 찾는다:
- 있으면 **교체**
- 없으면 "## 요약" 섹션 앞에 **추가**

### 스냅샷 내용

```markdown
## 📊 최근 상태 스냅샷

> 마지막 업데이트: 2026-04-30 14:20 (`/status --update`)

**진행률**: Story 8/18 (44%) · Task 28/52 (54%) · 크기 가중 46%

| 상태 | Story | Task |
|------|-------|------|
| ✅ 완료 | 8 | 28 |
| 🔄 진행중 | 3 | 7 |
| ⚠️ 블로킹 | 1 | 2 |
| ⬜ 예정 | 6 | 15 |

**크리티컬 패스**: E2-S1 (진행중) — 지연 시 M2 영향
**블로킹 이슈**: 1건 (E2-S1-T3: SES 승인 대기)
**다음 마일스톤**: M2 (3일 남음, 60% 달성)
```

`str_replace` 로 기존 스냅샷을 덮어쓰거나, 없으면 "## 요약" 앞에 삽입.

## 6단계: 주의사항

- Plan에 체크박스·상태 표기가 전혀 없으면 "진행 상황 표기가 없어 분석 불가" 안내하고 기본 체크박스 관례 설명
- 커스텀 상태 기호(`[~]`, `[!]`)는 **선택적** — 표준 `[ ]`/`[x]` 만 있어도 작동
- `--update` 는 **읽기 전용** Plan 파일을 수정하는 첫 경우 — 사용자에게 변경 내용 요약을 출력에 포함
- Plan 파일이 여러 개면 하나만 선택해 분석 (경로로 명시 필수)
- Plan 파싱 실패 시 가능한 한 부분 결과라도 출력하고 파싱 못한 부분 명시

## 상태 표기 가이드 (Plan 편집 시 참고)

사용자가 Plan을 편집하며 진행 상황을 남기려면:

```markdown
#### E1-S1: 토큰 테이블 마이그레이션 (DBA, M)
<!-- 상태: done -->

**Task**:
- [x] E1-S1-T1: users 테이블 컬럼 추가 마이그레이션 (dba, S)
- [x] E1-S1-T2: email_verification_tokens 테이블 생성 (dba, S)
- [~] E1-S1-T3: 온라인 마이그레이션 POC (dba, M)  ← 진행중
- [!] E1-S1-T4: 스테이징 적용 (dba, S)            ← 블로킹
- [ ] E1-S1-T5: 프로덕션 적용 (dba, S)             ← 예정
```
